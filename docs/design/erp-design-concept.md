# Daebang ERP Design Concept

작성일: 2026-06-06

## 목적

대방동 지역주택조합 ERP는 총회관리, 홈페이지, 조합원관리, 토지관리, 수지분석을 하나의 제품처럼 연결하는 웹 프로그램이다. 디자인 목표는 기존 지역주택조합 시스템의 무겁고 칙칙한 인상을 줄이고, 조합 사무국이 매일 쓰기 좋은 밝고 신뢰감 있는 업무 도구를 만드는 것이다.

기준 스타일은 루트의 `DESIGN.md`를 따른다. 다만 `DESIGN.md`의 원형은 넓은 여백과 장난스러운 일러스트가 강한 랜딩페이지 성격이므로, ERP 내부 화면에는 그대로 복사하지 않고 업무 밀도에 맞게 절제해서 적용한다.

## 제품 톤

핵심 톤은 "신뢰감 있는 친근함"이다.

- 밝고 환영감 있는 첫인상
- 조합원과 사무국 모두에게 부담 없는 색감
- 중요한 금액, 의결, 토지 상태는 명확하게 읽히는 업무용 밀도
- 캐주얼한 일러스트는 홈페이지와 빈 상태 화면에 집중
- 운영 화면은 장식보다 표, 필터, 상태, 이력 확인을 우선

피해야 할 톤은 다음과 같다.

- 관공서 문서처럼 회색 위주의 무거운 화면
- 부동산 분양 사이트처럼 과한 금색, 남색, 고급 이미지 중심 화면
- 스타트업 랜딩페이지처럼 실제 업무보다 장식이 앞서는 화면
- 조합원 정보, 분담금, 의결 결과를 가볍게 보이게 만드는 과한 캐릭터 사용

## 디자인 원칙

1. 홈페이지는 환영감, ERP는 명료함을 우선한다.
2. 같은 색상 토큰을 쓰되, 사용 강도를 화면 성격에 맞게 조절한다.
3. 조합원, 회계/자금, 총회, 토지, 수지분석은 각각 독립 모듈처럼 보이되 같은 내비게이션과 컴포넌트 규칙을 공유한다.
4. 중요한 업무 데이터는 표, 상태 배지, 필터, 상세 패널로 빠르게 스캔되게 한다.
5. 일러스트는 분위기를 여는 역할만 하고, 의사결정 UI를 대신하지 않는다.

## 색상 적용

`DESIGN.md`의 토큰을 기본 팔레트로 유지한다.

| 역할 | 색상 | ERP 적용 |
| --- | --- | --- |
| Page canvas | `#ebf5ff` Sky Wash | 홈페이지와 포털 배경의 기본. ERP 내부는 너무 넓게 쓰지 않고 사이드바 주변, 상단 배경, 빈 상태 영역에 사용 |
| Surface | `#fafdff` Paper White | 카드, 패널, 테이블 컨테이너의 기본 표면 |
| Primary text | `#0a0d12` Midnight Ink | 제목, 주요 수치, 테이블 핵심 값 |
| Primary action | `#181d27` Pressed Charcoal | 저장, 확정, 생성 같은 최상위 액션 버튼 |
| Secondary text/border | `#535862` Stone | 본문, 보조 라벨, 헤어라인 보더 |
| Muted text | `#93979f` Fog | 도움말, 비활성, 부가 정보 |
| Accent blue | `#4fbeff` Cornflower | 안내, 링크성 강조, 조합원 포털의 장식 포인트 |
| Accent orange | `#f26110` Tangerine | 주의, 미납, 마감 임박 등 따뜻한 경고 |
| Accent purple | `#9552e0` Amethyst | 총회, 의결, 서면결의 계열 포인트 |
| Accent mustard | `#bb9915` Mustard | 회계/자금, 수지분석, 예산, 추정치 계열 포인트 |

기능 상태 색상은 장식 팔레트와 섞어 쓰되 의미를 고정한다.

- 정상/완료: 부드러운 녹색 계열
- 진행중: Cornflower 또는 Signal Blue 계열
- 주의/미납: Tangerine 계열
- 보류/검토: Mustard 계열
- 오류/반려: 별도 Red 계열을 추가하되 과하게 쓰지 않는다

## 타이포그래피

한국어 웹 ERP이므로 기본 폰트는 Pretendard를 사용한다. `DESIGN.md`의 Aeonik/Geist 구분은 다음처럼 변환한다.

| 원형 | ERP 변환 |
| --- | --- |
| Aeonik display | Pretendard 700 또는 800, 랜딩페이지 대제목에 사용 |
| Geist UI/body | Pretendard 400, 500, 600, ERP 본문과 UI에 사용 |

권장 크기:

- 홈페이지 히어로 제목: 56-72px desktop, 34-44px mobile
- ERP 페이지 제목: 24-32px
- 카드 제목: 18-20px
- 테이블 본문: 13-14px
- 보조 텍스트: 12-13px
- 버튼: 14-15px

ERP 내부에서는 `DESIGN.md`의 음수 letter-spacing을 강하게 쓰지 않는다. 한글 가독성을 위해 letter-spacing은 기본값 또는 아주 약한 조정만 사용한다.

## 화면별 적용 방향

### 홈페이지

홈페이지는 `DESIGN.md`를 가장 강하게 적용한다.

- Sky Wash 배경
- 중앙 정렬 히어로 카피
- 조합원을 환영하는 밝은 일러스트
- 넓은 여백
- 둥근 CTA 버튼
- 조합 소개, 사업 진행, 공지, 자료실로 이어지는 카드형 섹션

히어로 이미지는 중앙 텍스트 영역을 비우고 양쪽이나 하단에 인물, 아파트, 주변 오브젝트를 배치한다. 일러스트는 젊은 남녀뿐 아니라 중년 남녀도 포함해 조합 구성원의 다양성을 보여준다.

### 조합원 포털

조합원 포털은 홈페이지보다 업무성이 높지만, 사무국 ERP보다는 부드럽게 간다.

- 내 정보, 납부 현황, 공지, 자료실, 문의를 탭 또는 카드로 구성
- 큰 숫자와 상태 배지로 내 납부/서류 상태를 바로 확인
- 안내 문구와 빈 상태 화면에 일러스트 사용
- 민감한 금액과 개인정보는 장식보다 명확한 구획을 우선

### ERP 내부 관리자

내부 관리자 화면은 밝은 SaaS 업무 도구로 잡는다.

- 좌측 사이드바 또는 상단-좌측 혼합 내비게이션
- 업무 영역은 Paper White 표면과 얇은 보더 중심
- 카드 radius는 8-16px 위주로 낮춘다
- 대시보드 요약 카드만 20-24px 정도의 큰 radius 허용
- 테이블, 필터, 검색, 상태 배지, 상세 드로어를 기본 조작 패턴으로 사용
- 큰 장식 일러스트는 대시보드 빈 상태, 온보딩, 오류 화면 정도로 제한

## 모듈별 성격

### 대시보드

ERP 첫 화면은 전체 사업 현황을 빠르게 판단하는 관제 화면이다.

- 조합원 수, 납부율, 미납 건수, 예산 집행률, 가용 자금, 토지 확보율, 총회 예정, 수지분석 최신 결과를 요약
- 각 카드에서 해당 모듈로 바로 이동
- 위험 항목은 별도 "주의 필요" 리스트로 제공

### 조합원관리

사람과 권리 관계를 관리하는 핵심 모듈이다.

- 밀도 높은 테이블 중심
- 조합원 상태, 계약 상태, 납부 상태, 서류 상태를 배지로 표시
- 상세 화면은 기본 정보, 계약/권리, 납부, 문서, 상담 이력을 탭으로 분리

### 회계/자금관리

조합 운영 자금과 사업비 집행을 관리하는 재무 통제 모듈이다. 조합원 납부 관리는 조합원 상세와 연결되지만, 회계/자금관리는 입출금 흐름, 계정과목, 예산 대비 집행, 신탁/대행/시공 관련 지출을 별도 원장으로 다룬다.

- 입출금 내역, 계정과목, 거래처, 증빙 문서를 한 화면에서 확인
- 예산 대비 집행률, 잔여 예산, 가용 자금, 미승인 지출을 KPI로 표시
- 신탁사, 업무대행사, 시공사, 설계/감리 등 주요 지출처를 분류
- 지출 요청, 검토, 승인, 지급완료 상태를 단계형으로 관리
- 금액 데이터는 우측 정렬하고, 변경/승인 이력은 감사 로그에 남긴다

### 총회관리

VoteCast와 연동되는 의사결정 모듈이다.

- 총회 목록, 안건, 출석, 위임, 서면결의, 투표 결과를 단계형으로 표시
- Amethyst 계열 포인트로 총회/의결 영역을 식별
- 확정된 의결 결과는 수정 가능 상태와 확정 상태를 명확히 구분

### 토지관리

db-landon과 연동되는 필지/소유자/확보율 모듈이다.

- 지도 또는 필지 목록을 중심으로 구성
- 확보, 협의중, 미계약, 등기완료 같은 상태를 명확히 구분
- 소유자 정보와 조합원 정보가 연결될 수 있으므로 식별자 매핑 UI가 필요

### 수지분석

valueON과 연동되는 사업성 판단 모듈이다.

- 시나리오 카드와 비교 테이블 중심
- Mustard 계열 포인트로 예산/추정 영역을 식별
- 최신 분석 결과, 기준일, 작성자, 가정값 변경 이력을 항상 노출

### 홈페이지/공지/문서

dbapt-site와 연결될 공개/포털 콘텐츠 영역이다.

- 홈페이지 콘텐츠와 ERP 문서 관리가 완전히 섞이지 않게 분리
- 공지, 자료실, FAQ는 조합원 포털과 연동 가능하게 설계
- 관리자 화면에서는 작성, 공개 상태, 열람 대상, 게시 기간을 명확히 제어

## 컴포넌트 규칙

### 버튼

- Primary: `#181d27` 배경, 흰색 텍스트
- Secondary: Paper White 배경, Stone 보더
- Destructive: 별도 red 계열, 확인 모달 필수
- Icon button: lucide 아이콘 사용, 툴팁 제공

### 카드와 패널

- 홈페이지: 24-32px radius 허용
- ERP 대시보드: 16-24px radius
- ERP 테이블/필터 패널: 8-12px radius
- 카드 안에 카드 중첩은 피하고, 필요하면 섹션/탭/드로어로 분리

### 테이블

- 행 높이 44-52px
- 고정 헤더와 컬럼 정렬 지원
- 금액은 우측 정렬
- 상태는 색상 배지와 텍스트를 함께 표시
- 대량 업무를 위해 검색, 필터, 정렬, 엑셀 내보내기를 기본 제공

### 배지

배지는 색상만으로 의미를 전달하지 않는다. 항상 텍스트를 포함한다.

- 납부완료
- 미납
- 연체
- 검토중
- 확정
- 공개
- 비공개

### 폼

- 긴 조합원 등록 폼은 단계형 또는 섹션 접기 방식 사용
- 필수 항목, 검증 오류, 저장 상태를 명확히 표시
- 개인정보와 금액 입력은 입력 마스크와 확인 요약을 제공

## 레이아웃 기준

홈페이지:

- 최대 폭 1200px
- 섹션 간격 64-80px
- 히어로는 첫 화면에서 브랜드와 목적이 바로 보이게 구성
- 다음 섹션 일부가 첫 화면 하단에 살짝 보이도록 높이를 잡는다

ERP 내부:

- 좌측 내비게이션 폭 240-280px
- 상단 바 높이 56-64px
- 콘텐츠 최대 폭은 화면에 맞게 유동적으로 확장
- 테이블 페이지는 1200px 제한을 두지 않는다
- 상세 정보는 우측 드로어 또는 별도 상세 페이지로 제공

## 연동 UI 방향

기존 5개 앱은 ERP 안에서 "연동 모듈"로 보이게 한다.

| 앱 | ERP 표시 이름 | UI 성격 |
| --- | --- | --- |
| VoteCast | 총회관리 | 단계형 진행, 의결 확정, 결과 보존 |
| dbapt-site | 홈페이지/포털 | 공개 콘텐츠, 조합원 안내, 공지 |
| peopleON | 조합원관리 | 원장, 권리, 정산, SMS |
| ERP Core/peopleON | 회계/자금 | 입출금, 계정과목, 예산 집행, 주요 지출 |
| db-landon | 토지관리 | 필지, 소유자, 확보율 |
| valueON | 수지분석 | 사업성, 시나리오, 추정치 |

ERP에는 각 모듈의 "연동 상태"를 보여주는 공통 패턴이 필요하다.

- 마지막 동기화 시각
- 데이터 출처
- 연결 상태
- 오류 발생 시 원인과 재시도 버튼
- 외부 앱으로 이동하는 링크

## 보안과 신뢰 UX

조합 ERP는 데이터 신뢰가 디자인의 일부다.

- 중요한 변경은 저장 전 요약 확인
- 확정된 총회 결과, 납부 데이터, 입출금/지출 승인 데이터는 감사 로그를 남긴다
- 관리자 권한과 조합원 권한의 화면 경계를 명확히 한다
- 개인정보, 납부, 의결 결과는 접근 권한을 시각적으로 드러낸다
- 삭제보다는 비활성화, 보관, 취소 이력을 우선한다

## 1차 화면 우선순위

1. ERP 대시보드
2. 조합원 목록/상세
3. 회계/자금 요약
4. 총회 연동 요약
5. 토지 확보율 요약
6. 수지분석 요약
7. 문서/공지 관리
8. 연동 설정

## 구현 메모

- 기술은 Next.js + shadcn/ui를 기본 후보로 둔다.
- 색상 토큰은 `DESIGN.md`의 CSS custom properties를 기반으로 확장한다.
- 한국어 UI는 Pretendard를 기본 폰트로 쓴다.
- ERP 내부 화면은 랜딩페이지보다 낮은 radius와 높은 정보 밀도를 사용한다.
- 실제 구현 전에는 Supabase 키 노출 여부와 환경변수 관리를 먼저 정리한다.

## 결정 사항

- 디자인 기준 문서: `DESIGN.md`
